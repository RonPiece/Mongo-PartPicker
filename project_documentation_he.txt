תיעוד פרויקט Mongo-PartPicker (Project Documentation)

סקירת פרויקט (Project Overview)
מערכת MongoDB מקיפה לניהול קטלוג רכיבי מחשב ובניית מפרטי מחשב מותאמים אישית. הפרויקט מדגים עקרונות MongoDB מתקדמים כולל Polymorphic Pattern, ETL data transformation, וצינורות עיבוד נתונים מורכבים (Aggregation Pipelines).

---

1. תכנון ואפיון (Data Modeling - Section 1)

1.1 אפיון כללי של מסד הנתונים (General Characterization)
המערכת מתוכננת לנהל קטלוג של רכיבי מחשב ומפרטים (Builds) שנוצרו על ידי משתמשים. מבנה ה-Database עבר אופטימיזציה הן עבור פעולות קריאה (Read-heavy) כמו צפייה ברכיבים וסקירות, והן עבור פעולות כתיבה (Write-heavy) כמו יצירת הזמנות ושמירת מפרטים.

1.2 החלטה על מבנה ה-Database (References vs. Embedded)
המלצה: המטלה ממליצה לשלב בין Embedded ל-Referenced. יישמנו גישה היברידית (Hybrid Approach) הממקסמת את היתרונות של שתי השיטות:

אוסף הרכיבים (Components Collection):
תבנית שנבחרה: Polymorphic
סיבת ההחלטה: אנו משתמשים באוסף `components` יחיד לכל הסוגים (CPU, GPU, וכו') מכיוון שהם חולקים שדות משותפים (`price`, `name`, `manufacturer`) אך בעלי תת-מסמך `specs` ייחודי לכל סוג. זה מפשט חיפושים רוחביים בקטלוג.

ביקורות בתוך רכיב (Reviews):
תבנית שנבחרה: Embedded
סיבת ההחלטה: בעת הצגת מוצר, אנו תמיד צריכים את הביקורות שלו. הטמעה (Embedding) חוסכת פעולת `$lookup` נוספת ומשפרת ביצועי קריאה.

אוסף המפרטים (Builds Collection):
תבנית שנבחרה: Referenced
סיבת ההחלטה: מפרטים מפנים לרכיבים באמצעות `ObjectId` במערך `parts`. מדוע? רכיבים הם ישויות נפרדות הקיימות בפני עצמן. אם היינו מטמיעים את הרכיבים, עדכון מחיר היה דורש עדכון של אלפי מפרטים. הפניות מבטיחות עקביות נתונים (Data Consistency).

אוסף המשתמשים (Users Collection):
תבנית שנבחרה: Hybrid
- הזמנות (Orders) הן Embedded: היסטוריית הזמנות היא סטטית ושייכת בלעדית למשתמש.
- מפרטים שמורים (Saved Builds) הם Referenced: המשתמשים מפנים למפרטים כך שאם מפרט מתעדכן, המשתמש רואה את השינוי.

1.3 ויזואליזציה של הסכמה (Schema Visualization)
ראה את קטע הקוד ב-project_documentation.md עבור דוגמאות ב-JavaScript.

2. מיפוי דרישות: סעיף 3 (Requirement Mapping: Section 3 JSON & JS)

דרישה: "כתיבת קטעי קוד ופונקציות ב-JSON לצרכים שונים, גם בהקמה וגם בשליפות."
דרישה זו מיושמת לאורך כל הקוד בשני הקבצים:

Setup & ETL (בקובץ data.js):
8 פונקציות טרנספורמציה שונות (`cpuToComponent` וכו') הממירות JSON גולמי למסמכי MongoDB.

לוגיקה מורכבת (Complex Logic בקובץ data.js):
הפונקציה `detectCpuSocket` - אלגוריתם מתקדם לפענוח מחרוזות לקביעת תאימות תושבת ה-CPU.

פונקציות עזר (Utility בקובץ data.js):
פונקציות `safeNumber` ו-`normalizeSpaces` לניקוי נתונים.

כלי הוספה (Insert Utility בקובץ project.js):
הפונקציה `batchInsert` - כלי עזר להוספה יעילה של מסות נתונים ל-Database.

אגרגציה (Aggregation בקובץ project.js):
הפונקציה `autoBuilderPipeline` - פונקציית JavaScript שמייצרת באופן דינמי Aggregation Pipeline מורכב.

MapReduce (בקובץ project.js):
פונקציות `map`, `reduce`, ו-`finalize` כתובות ב-JavaScript טהור כנדרש ב-MapReduce.

3. הסבר על ארכיטקטורת שני הקבצים (Two-File Architecture Explanation)

מדוע לפצל ל-data.js ו-project.js?
הפיצול הוא מכוון ומספק יתרונות משמעותיים:

קובץ data.js (~27KB קוד):
טוען קבצי JSON (בסך הכל ~7MB), מספק מערכים גולמיים, ומטפל בלוגיקה של ה-ETL (Extract, Transform, Load).

קובץ project.js (~66KB קוד):
מטפל בפעולות Database: הקמה, הוספות, שאילתות, עדכונים, ו-Aggregation.

הבעיה שנפתרה:
קבצי ה-JSON הגולמיים שוקלים יחד כ-7MB. הטמעה ישירה שלהם בקוד הייתה תוקעת את רוב ה-IDEs.
הפתרון: `data.js` טוען את הקבצים הללו בזמן ריצה (Runtime) באמצעות `cat()` + `JSON.parse()`.

4. הוראות הרצה (How to Run)

ב-MongoDB Shell (mongosh):
1. נווט לתיקייה: `cd("path/to/Mongo-PartPicker")`
2. טען את הקובץ הראשי: `load("project.js")` (זה טוען אוטומטית גם את `data.js`)

פונקציות לבדיקה (הרצה ידנית):
- `section4_findAndQuery()`: מריץ 7 שאילתות מגוונות.
- `section5_updatesAndDeletes()`: מריץ עדכונים, מחיקות, ופעולות על אוספים.
- `section6_aggregations()`: מריץ Aggregation Pipelines מורכבים.
- `section7_mapReduce()`: מריץ 4 דוגמאות MapReduce.

---

5. תהליך טרנספורמציית הנתונים (ETL Data Transformation Process)

התהליך: Extract (חילוץ) -> Transform (המרה) -> Load (טעינה)
כל סוג רכיב עובר דרך פונקציית טרנספורמציה ייעודית ב-`data.js`. לדוגמה, הפונקציה `cpuToComponent` מחשבת ציון ביצועים ומזהה את סוג התושבת (Socket) באופן אוטומטי מתוך שם המעבד.

---

6. נקודות חוזק ופוטנציאל לציון (Grade Potential & Strengths)

הפרדת רשויות (Separation of Concerns):
לוגיקה מפוצלת בצורה נקייה בין הכנת נתונים (data.js) לפעולות DB (project.js).

יישום סעיף 3 (Section 3 Implementation):
שימוש נרחב בלוגיקה של JavaScript עבור ETL, יצירת Pipelines, ו-MapReduce.

תבנית פולימורפית (Polymorphic Pattern):
תכנון MongoDB מקצועי המטפל בסוגי רכיבים מגוונים באוסף אחד.

לוגיקה לזיהוי תושבת (Socket Detection Logic):
עבודה אלגוריתמית מרשימה בפונקציה `detectCpuSocket`.

Embedded + Referenced:
תכנון היברידי בדיוק כפי שהתבקש בסעיף 1.
