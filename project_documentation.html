סעיף 6 הוא הלב של הפרויקט ומכיל מערכת אינטראקטיבית מתקדמת להרכבת מפרט מחשב, לצד שני צינורות אנליטיים.

8.1 Pipeline #1: Interactive PC Builder — בניית מחשב אינטראקטיבית

המערכת מאפשרת למשתמש לבנות מפרט מחשב מלא בשמונה שלבים, כאשר בכל שלב מוצגים רק רכיבים תואמים ותקציביים:

שלבי הבנייה:

שלברכיבטכניקת MongoDBלוגיקה מיוחדת
1CPUSection 6: aggregate עם $match, $project, $sort, $limitסינון רק Sockets מודרניים (AM4, AM5, LGA1700...)
2MotherboardSection 6: $lookup Self-Join + $unwind + $match + $projectTier Enforcement: לוח-אם חייב להיות בטווח ±1 Tier מה-CPU
3RAMSection 4: find().sort().limit()סינון DDR4/DDR5 אוטומטי לפי לוח-האם. סינון ECC/Server RAM
4GPUSection 4: find עם סינון ציון וגודלדורש length_mm — בלי זה שלב המארז לא יכול לבדוק התאמה. Surplus Tax: 70% מהחיסכון בשלבים קודמים
5StorageSection 4: find עם סינון סוגSSD בלבד כאחסון ראשי. טווח קיבולת לפי פרופיל
6CPU CoolerSection 4: find עם סינון TDPחסימת מקררים Low-Profile למעבדים חזקים. מגבלת radiator למארזים קטנים
7PSUSection 3: calculateRequiredWatts() + findTier Enforcement: ספק כח חייב להתאים לטיר ה-GPU
8CaseSection 6: aggregate עם $subtractבדיקת GPU clearance (מקום פיזי). בדיקת Form Factor תאימות

שלב ה-$lookup (Self-Join) — הסבר מפורט:

בשלב הלוח-אם, המערכת מבצעת $lookup Self-Join: היא מחפשת באותו אוסף (components) לוחות-אם שה-socket שלהם תואם ל-socket של המעבד שנבחר:

Stageפקודהמה עושה
1$matchמוצא את המעבד שנבחר לפי _id
2$lookupSelf-Join: from: "components", localField: "requirements.socket_match", foreignField: "specs.socket" — מצרף רכיבים עם אותו socket
3$unwindשוטח את מערך התוצאות (socket_matches) — כל התאמה הופכת למסמך נפרד
4$matchמסנן רק לוחות-אם (type = "Motherboard") בטווח מחירים ועם ram_type תקין
5$projectמעצב את התוצאה לפורמט אחיד (שם, מחיר, מפרטים)
6-7$sort + $limitממיין לפי מחיר (זול קודם), מגביל ל-30

לוגיקת הקצאת תקציב דינמית:

בכל שלב, המערכת מחשבת:

  • maxPrice = min(budget × profileCap, availableBudget - futureReserve)
  • futureReserve = סכום העלויות המינימליות של כל הרכיבים שעוד לא נבחרו

כך מובטח שתמיד יישאר מספיק תקציב לרכיבים הנותרים.

מנגנון Tier Enforcement:

  • ה-buildTier נקבע לפי הטיר של ה-CPU (הרכיב הראשון)
  • כל רכיב עוקב מקבל רצפת מחיר מינימלית: tierToFloor(buildTier - 1, tierBounds)
  • לדוגמה: אם ה-CPU הוא Tier "High" (טיר 3), כל רכיב תמיכה חייב להיות Tier "Mid" (טיר 2) לפחות
  • מנגנון Fallback: אם התקציב לא מספיק לטיר הנדרש, המערכת יורדת טיר ומדפיסה אזהרה

שלב הסיום (finalizeBuild):

  • Section 3: מחשב calculateWeightedScore
  • Section 4: insertOne — שומר את המפרט ב-recommended_combos
  • Section 6: aggregate עם $add — MongoDB מחשב את סכום המחירים באופן עצמאי (אימות)
  • Section 6: $out — שומר את התוצאה בחזרה ל-recommended_combos

9. סעיף 7: MapReduce — ניתוחים סטטיסטיים

סעיף זה מכיל 3 ניתוחי MapReduce שונים:

9.1 MapReduce A: Best Builds Per Tier — section7_mapReduce()

ניתוח מורכב שמוצא את המפרט הטוב ביותר בכל טווח מחירים.

תהליך העבודה:

  1. שלב ההכנה: מייצר עשרות מפרטים אוטומטיים (buildComputerByBudgetDoc) — מריץ את כל שלבי ה-PC Builder בשקט (ללא הדפסה) עבור 4 פרופילים × n תקציבים
  2. שלב הגבולות: מחשב Tier Boundaries דינמיים מהמחירים בפועל (לא מהתקציב המבוקש), באמצעות aggregate עם $min ו-$max
  3. Map: מקבל כל מפרט, קובע לאיזה Tier תקציבי הוא שייך (לפי total_price), ושולח emit(tier, buildData)
  4. Reduce: משווה מפרטים באותו Tier ובוחר את זה עם ה-score הגבוה ביותר (ואם שווה — הזול ביותר)
  5. Finalize: מעצב את הפלט עם שם הזוכה, ציון, מחיר, ורשימת הרכיבים
רכיב MapReduceתיאור
Map FunctionmapBudgetTier — מחלק לטיירים דינמיים לפי total_price. משתמש ב-scope variable (_tierBounds_)
Reduce FunctionreduceBestBuild — בוחר את הציון המקסימלי. ב-tie — בוחר את הזול
Finalize FunctionfinalizeBestBuild — מעצב: budget_tier, winner, performance_score, actual_price, components
Query Filter{total_price: {$type: "number"}}
Scope{_tierBounds_: tierBounds} — מעביר את גבולות ה-Tier לתוך פונקציית ה-Map
Output Collectionbest_builds_per_tier
