5. סעיף 3: פונקציות JavaScript — עיבוד, חישובים ולוגיקה עסקית

סעיף זה מכיל את כל הפונקציות שנכתבו ב-JavaScript ומשמשות את המערכת לאורך כל הפרויקט. הפונקציות מחולקות לקטגוריות לפי ייעודן.

5.1 פונקציות ETL (בקובץ data.js)

פונקציות אלו אחראיות על תהליך ה-Transform — המרת הנתונים הגולמיים מ-JSON למבנה הפולימורפי של MongoDB.

פונקציה תיאור דוגמה
detectCpuSocket(cpuName) זיהוי אוטומטי של סוג Socket מתוך שם המעבד באמצעות Regex. מזהה 15+ דפוסים שונים (LGA1700, AM5 וכו') "Intel Core i9-14900K" → "LGA1700"
cpuToComponent(raw) ממיר אובייקט CPU גולמי למבנה הפולימורפי. מחשב score = cores×100 + base_clock×50 אובייקט גולמי → מסמך MongoDB מוכן
gpuToComponent(raw) ממיר GPU גולמי. מחלץ chipset, vram, length_mm לצורך בדיקות תאימות מארז כולל חישוב score = vram × 200
motherboardToComponent(raw) ממיר לוח-אם. כולל inferMotherboardRamType שקובע DDR4/DDR5 לפי socket AM5 → DDR5, LGA1200 → DDR4
ramToComponent(raw) ממיר זיכרון RAM. מנרמל שם לפורמט אחיד (שם + DDR + מהירות)
storageToComponent(raw) ממיר אחסון. מנרמל קיבולת (2000GB → "2TB")
cpuCoolerToComponent(raw) ממיר מקרר. מחלץ סוג (Air/Liquid), רעש, RPM
psuToComponent(raw) ממיר ספק כח. מחלץ הספק, דירוג יעילות
normalizeSpaces(s) ניקוי רווחים כפולים מטקסט
safeNumber(value, fallback) המרה בטוחה למספר עם ערך ברירת מחדל
detectGpuManufacturer(chipset) מזהה יצרן GPU מתוך שם הצ'יפסט באמצעות חיפוש מחרוזות (GeForce→NVIDIA, Radeon→AMD, Arc→Intel) "GeForce RTX 4090" → "NVIDIA"
normalizeGpuChipset(chipset) מנרמל שם צ'יפסט — מתקן עיצוב טקסט (SUPER → Super)
normalizeMotherboardName(name) מנרמל שמות לוחות-אם לפורמט אחיד (Asus→ASUS, WIFI→WiFi, HERO→Hero)
inferMotherboardRamType(socket, boardName) קובע סוג זיכרון (DDR4/DDR5) לפי socket ושם הלוח. בודק תחילה שם מפורש, אחר כך כללי socket AM5 → DDR5, LGA1200 → DDR4
cleanCaseFormFactor(caseType) ניקוי והמרת Form Factor של מארזים לפורמט אחיד
inferCaseSupportedMotherboards(caseType) מסיק מערך לוחות-אם נתמכים לפי סוג המארז. היררכיה: ATX תומך גם ב-Micro ATX וב-Mini ITX
normalizeRamName(rawName, ddrGen, speedMhz) בניית שם RAM אחיד בפורמט: שם + DDR + מהירות "Trident Z5 RGB", 5, 6400 → "Trident Z5 RGB DDR5-6400"
normalizeStorageName(rawName, capacityGb) המרת גדלי אחסון ליחידות קריאות (2000GB → "2TB")
transformArray(rawArr, transformFn) פונקציית עיבוד גנרית — מפעילה פונקציית transformer על כל אלמנט במערך
loadJsonArray(path) טעינת קובץ JSON מהדיסק באמצעות cat() + JSON.parse(), כולל validation שהתוצאה היא מערך
resolveMppPath(relPath) פתרון נתיבי קבצים — משתמש במשתנה גלובלי __MPP_REPO_ROOT__ אם קיים, אחרת נתיב יחסי

רציונל: כל פונקציית Transformer מייצרת מסמך אחיד עם שדות type, name, manufacturer, price, specs ו-requirements. כך ניתן לשמור 8 סוגי רכיבים שונים באוסף אחד (Polymorphic Pattern) תוך שמירה על יכולת שליפה אחידה.

5.4 פונקציות ציון ביצועים וחישוב תאימות

פונקציה מה מחשבת נוסחה / לוגיקה
getProfileParams(usageType) מחזירה פרמטרים עבור פרופיל שימוש: gaming, workstation, budget, enthusiast כל פרופיל מגדיר: יחסי הקצאת תקציב לכל רכיב (cpuCap, gpuCap...), טווח RAM מינימלי/מקסימלי, ומשקלות ציון (למשל ל-gaming: gpu:0.6, cpu:0.4, ram:0.0)
calculateWeightedScore(cpu, gpu, ram, weights) ציון ביצועים משוקלל cpuScore × weights.cpu + gpuScore × weights.gpu + ramCapacity × weights.ram
calculateRequiredWatts(cpuTdp, gpuVram, cpuScore) דרישת הספק מינימלית לספק כח (effectiveCPU + gpuVram×20) × 1.25 + 50W מרווח. מתקנת TDP רשמי מטעה (i9-12900F רשום 65W אבל שואב 200W+ בפועל)
getCoolerReserve(tdp, cpuScore) מחיר מינימלי למקרר בהתאם ל-TDP TDP ≥150W→$80, ≥125W→$60, ≥95W→$40, >65W→$25, אחרת→$15

רציונל פונקציות הציון: בעולם האמיתי, מעבדים רבים רושמים TDP נמוך (65W) אך צורכים 200W+ בפועל. הפונקציה calculateRequiredWatts משתמשת ב-score כגורם מתקן: מעבד עם ציון מעל 1500 מקבל TDP אפקטיבי של 150W לפחות, ומעבד עם ציון מעל 2000 מקבל 200W. כך מובטח שספק הכח יספיק בפועל.

5.5 מערכת ה-Tier — דירוג איכות מבוסס מחיר

המערכת כוללת מנגנון Tier חכם שמבטיח שמפרט יקר לא ילווה ברכיבי תמיכה זולים ולא מתאימים:

פונקציה תיאור
getTierBoundaries(type) מחשבת גבולות Tier דינמיים לפי אחוזוני מחיר ב-DB:
Entry (0–25%), Mid (25–60%), High (60–85%), Enthusiast (85%+)
משתמשת ב-aggregate עם $arrayElemAt ו-$floor
priceTier(price, bounds) מחזירה מספר Tier (1–4) לפי מחיר וגבולות
tierToFloor(minTier, bounds) מחזירה מחיר מינימלי (רצפה) עבור Tier נתון

רציונל: בלי מנגנון ה-Tier, מפרט בתקציב $4000 עלול לקבל ספק כח של $30 או מקרר של $15 — רכיבים נמוכי איכות שלא מתאימים למערכת יוקרתית. ה-buildTier נקבע לפי הטיר של ה-CPU (הרכיב הראשון שנבחר) ומשפיע על המחיר המינימלי של כל שאר הרכיבים.

8. סעיף 6: צינורות אגרגציה — Interactive PC Builder ואנליטיקה

סעיף 6 הוא הלב של הפרויקט ומכיל מערכת אינטראקטיבית מתקדמת להרכבת מפרט מחשב, לצד שני צינורות אנליטיים.

8.1 Pipeline #1: Interactive PC Builder — בניית מחשב אינטראקטיבית

המערכת מאפשרת למשתמש לבנות מפרט מחשב מלא בשמונה שלבים, כאשר בכל שלב מוצגים רק רכיבים תואמים ותקציביים:

שלבי הבנייה:

שלב רכיב טכניקת MongoDB לוגיקה מיוחדת
1 CPU Section 6: aggregate עם $match, $project, $sort, $limit סינון רק Sockets מודרניים (AM4, AM5, LGA1700...)
2 Motherboard Section 6: $lookup Self-Join + $unwind + $match + $project Tier Enforcement: לוח-אם חייב להיות בטווח ±1 Tier מה-CPU
3 RAM Section 4: find().sort().limit() סינון DDR4/DDR5 אוטומטי לפי לוח-האם. סינון ECC/Server RAM
4 GPU Section 4: find עם סינון ציון וגודל דורש length_mm — בלי זה שלב המארז לא יכול לבדוק התאמה. Surplus Tax: אם נחסך תקציב בשלבים קודמים (CPU + Mobo + RAM), 70% מהחיסכון מופנה ל-GPU כדי למקסם ביצועים גרפיים
5 Storage Section 4: find עם סינון סוג SSD בלבד כאחסון ראשי. טווח קיבולת לפי פרופיל
6 CPU Cooler Section 4: find עם סינון TDP חסימת מקררים Low-Profile למעבדים חזקים. מגבלת radiator למארזים קטנים
7 PSU Section 3: calculateRequiredWatts() + find Tier Enforcement: ספק כח חייב להתאים לטיר ה-GPU
8 Case Section 6: aggregate עם $subtract בדיקת GPU clearance (מקום פיזי). בדיקת Form Factor תאימות

שלב ה-$lookup (Self-Join) — הסבר מפורט:

בשלב הלוח-אם, המערכת מבצעת $lookup Self-Join: היא מחפשת באותו אוסף (components) לוחות-אם שה-socket שלהם תואם ל-socket של המעבד שנבחר:

Stage פקודה מה עושה
1 $match מוצא את המעבד שנבחר לפי _id
2 $lookup Self-Join: from: "components", localField: "requirements.socket_match", foreignField: "specs.socket" — מצרף רכיבים עם אותו socket
3 $unwind שוטח את מערך התוצאות (socket_matches) — כל התאמה הופכת למסמך נפרד
4 $match מסנן רק לוחות-אם (type = "Motherboard") בטווח מחירים ועם ram_type תקין
5 $project מעצב את התוצאה לפורמט אחיד (שם, מחיר, מפרטים)
6-7 $sort + $limit ממיין לפי מחיר (זול קודם), מגביל ל-30

לוגיקת הקצאת תקציב דינמית:

בכל שלב, המערכת מחשבת:

  • maxPrice = min(budget × profileCap, availableBudget - futureReserve)
  • futureReserve = סכום העלויות המינימליות של כל הרכיבים שעוד לא נבחרו

כך מובטח שתמיד יישאר מספיק תקציב לרכיבים הנותרים.

מנגנון Tier Enforcement:

  • ה-buildTier נקבע לפי הטיר של ה-CPU (הרכיב הראשון)
  • כל רכיב עוקב מקבל רצפת מחיר מינימלית: tierToFloor(buildTier - 1, tierBounds)
  • לדוגמה: אם ה-CPU הוא Tier "High" (טיר 3), כל רכיב תמיכה חייב להיות Tier "Mid" (טיר 2) לפחות
  • מנגנון Fallback: אם התקציב לא מספיק לטיר הנדרש, המערכת יורדת טיר ומדפיסה אזהרה

שלב הסיום (finalizeBuild):

  • Section 3: מחשב calculateWeightedScore
  • Section 4: insertOne — שומר את המפרט ב-recommended_combos
  • Section 6: aggregate עם $add — MongoDB מחשב את סכום המחירים באופן עצמאי (אימות)
  • Section 6: $out — שומר את התוצאה בחזרה ל-recommended_combos

מנגנון Fallback מרובה שכבות (Defensive Programming):

בכל שלב של ה-PC Builder ממומש מנגנון fallback מרובה שכבות (2–3 fallbacks בכל שלב) המבטיח שהבנייה תמיד תצליח, גם בתקציבים קיצוניים:

  • Cooler: סינון לפי TDP מינימלי → חסימת Low-Profile למעבדים חזקים → מגבלת radiator למארזים קטנים → Fallback: הזול ביותר מעל minPrice
  • Case: סינון GPU clearance + Form Factor → Fallback 1: חיפוש מעל לתקציב → Fallback 2: סובלנות 15mm (tight fit) → Fallback 3: כל מארז תואם
  • PSU: Tier enforcement → Fallback: הורדת טייר → Fallback: הזול ביותר עם ההספק הנדרש

מנגנון זה מדגים Graceful Degradation — המערכת מורידה איכות במקום להכשל, תוך הדפסת אזהרות למשתמש.

אוסף דינמי — recommended_combos:

בנוסף ל-3 האוספים המרכזיים, המערכת יוצרת אוסף דינמי בשם recommended_combos שנוצר אוטומטית ע"י ה-PC Builder. אוסף זה מכיל את כל המפרטים שהורכבו (כולל פירוט רכיבים, ציוני ביצועים, ותאימות), ומשמש כקלט ל-MapReduce ב-Section 7.

8.2 Pipeline #2: Market Analysis — section6_marketAnalysis()

צינור אגרגציה שמייצר סטטיסטיקות שוק מלאות:

Stage פקודה מה עושה
1 $match מסנן רכיבים עם מחיר מספרי חיובי
2 $group מקבץ לפי type. מחשב: count ($sum), avg_price ($avg), min_price ($min), max_price ($max)
3 $project מוסיף est_market_value = $round($multiply(count, avg_price)). מעגל avg_price ל-2 ספרות באמצעות $round: ["$avg_price", 2]
4 $sort ממיין לפי count (יורד)

פלט: טבלה עם סטטיסטיקות מחיר לכל סוג רכיב — כמות, ממוצע, מינימום, מקסימום, ואומדן שווי שוק.

8.3 Pipeline #3: Manufacturer Breakdown — section6_manufacturerBreakdown()

צינור אגרגציה מורכב עם שתי שכבות $group:

Stage פקודה מה עושה
1 $match מסנן רכיבים עם יצרן ומחיר תקינים
2 $group (ראשון) מקבץ לפי {manufacturer, type}. מחשב: product_count ($sum), avg_price ($avg), products ($push)
3 $group (שני) מקבץ שוב לפי manufacturer. מחשב: total_products ($sum), categories ($push), first_category ($first)
4 $sort ממיין לפי total_products (יורד)
5 $limit מגביל ל-15 יצרנים

פלט: פריסת מוצרים לפי יצרן, כולל חלוקה לקטגוריות ומחיר ממוצע בכל קטגוריה.

9. סעיף 7: MapReduce — ניתוחים סטטיסטיים

סעיף זה מכיל 3 ניתוחי MapReduce שונים:

9.1 MapReduce A: Best Builds Per Tier — section7_mapReduce()

ניתוח מורכב שמוצא את המפרט הטוב ביותר בכל טווח מחירים.

תהליך העבודה:

  1. שלב ההכנה: מייצר עשרות מפרטים אוטומטיים (buildComputerByBudgetDoc) — מריץ את כל שלבי ה-PC Builder בשקט (ללא הדפסה) עבור 4 פרופילים × n תקציבים
  2. שלב הגבולות: מחשב Tier Boundaries דינמיים מהמחירים בפועל (לא מהתקציב המבוקש), באמצעות aggregate עם $min ו-$max
  3. Map: מקבל כל מפרט, קובע לאיזה Tier תקציבי הוא שייך (לפי total_price), ושולח emit(tier, buildData)
  4. Reduce: משווה מפרטים באותו Tier ובוחר את זה עם ה-score הגבוה ביותר (ואם שווה — הזול ביותר)
  5. Finalize: מעצב את הפלט עם שם הזוכה, ציון, מחיר, ורשימת הרכיבים
רכיב MapReduce תיאור
Map Function mapBudgetTier — מחלק לטיירים דינמיים לפי total_price. משתמש ב-scope variable (_tierBounds_)
Reduce Function reduceBestBuild — בוחר את הציון המקסימלי. ב-tie — בוחר את הזול
Finalize Function finalizeBestBuild — מעצב: budget_tier, winner, performance_score, actual_price, components
Query Filter {total_price: {$type: "number"}}
Scope {_tierBounds_: tierBounds} — מעביר את גבולות ה-Tier לתוך פונקציית ה-Map
Output Collection best_builds_per_tier

9.2 MapReduce B: Manufacturer Stats — section7_manufacturerStats()

ניתוח סטטיסטי ליצרנים — רק עבור רכיבים שמחירם מעל $200.

רכיב MapReduce תיאור
Map Function שולח emit(manufacturer, {count: 1, totalPrice: price})
Reduce Function סוכם count ו-totalPrice
Finalize Function מחשב avgPrice = totalPrice / count
Query Filter {price: {$gt: 200}} — רק רכיבים שמחירם מעל $200
Output Collection manufacturer_stats_mr

9.3 MapReduce C: Rating Distribution — section7_ratingDistribution()

ניתוח התפלגות דירוגי ביקורות — כמה ביקורות בכל ציון (1–5 כוכבים).

רכיב MapReduce תיאור
Map Function עובר על מערך reviews של כל רכיב, שולח emit(rating, 1) לכל ביקורת
Reduce Function Array.sum(values) — סיכום
Finalize Function מעצב: {rating, count}
Query Filter אין — עובר על כל הרכיבים (מתעלם מאלו ללא ביקורות)
Output Collection rating_distribution

10. החלטות עיצוב מרכזיות — שאלות ותשובות

סעיף זה מרכז את ההחלטות הטכניות המרכזיות בפרויקט והרציונל מאחוריהן:

נושא שאלה תשובה
getSiblingDB vs use למה getSiblingDB("MongoPartPicker") ולא use? use היא פקודת shell ולא עובדת מתוך סקריפט JavaScript ב-mongosh. getSiblingDB() הוא המקבילה הפרוגרמטית — מאפשר להחליף DB מתוך קוד.
$type: "number" למה מופיע price: { $type: "number" } בכל שאילתה? הנתונים אמיתיים מ-PCPartPicker — כ-36,000 רכיבים חסרי מחיר שמורים כ-null. ללא בדיקת $type, פקודות $sort ו-$gte יחזירו תוצאות שגויות (null נחשב קטן מכל מספר). זה Defensive Programming.
Idempotent Seed למה מוחקים את כל הנתונים (drop) בכל טעינה? מבטיח Idempotency — כל הרצה של load("project.js") מייצרת מצב זהה ונקי. מונע כפילויות ואנומליות.
patchDemoCriticalRawData() מה הפונקציה עושה ולמה נחוצה? הנתונים מ-PCPartPicker לא מובטח שכוללים RAM ו-Storage ספציפיים שה-demo data מחפש ב-findOne לפי שם מדויק. הפונקציה מוודאת שהנתונים הגולמיים מכילים את הרכיבים הקריטיים לדמו, ומסנתזת אותם אם חסרים — Data Integrity של ה-Seed.
pick() — State Machine איך ה-PC Builder יודע לאיזה שלב לעבור? pick() היא פונקציית Dispatcher שמשתמשת ב-buildState.step (State Machine גלובלי) כדי לקרוא לפונקציה הבאה ברצף. המשתמש לא צריך לזכור שמות פונקציות — רק pick(3).
הערה: mongosh הוא ES5 מוגבל (לא class, לא let/const). State גלובלי הוא הפתרון הפרקטי ביותר.
סובלנות תקציב 110% validateAndSave מאפשרת חריגה של 10% — למה לא 100%? בחירה מודעת: לפעמים אין רכיב בדיוק בתקציב. עדיף לאפשר build שעובד עם חריגה קטנה (10%) מאשר לחסום לחלוטין.
סדר שלבי הבנייה למה דווקא CPU → Mobo → RAM → GPU → Storage → Cooler → PSU → Case? הסדר הכרחי כי כל שלב תלוי בקודמו:
CPU קובע Socket → Mobo חייב אותו Socket → RAM חייב DDR תואם → GPU צריך length_mm לבדיקת Case → Cooler תלוי ב-TDP → PSU תלוי בכל הרכיבים → Case חייב להתאים לכולם.

להלן תרשים המתאר את הקשרים בין האוספים:

┌──────────────────────────────┐ │ components │ │ _id (ObjectId) │ │ type, name, manufacturer │ │ price, specs{}, requirements│ │ reviews[] (Embedded) │ │ price_history[] (Embedded) │ └──────────┬───────────────────┘ │ │ Referenced (ObjectId) │ ┌──────────┴───────────────────┐ ┌─────────────────────────────────────┐ │ builds │ │ users │ │ _id (ObjectId) │ │ _id (ObjectId) │ │ build_name, creator_name │ │ username, email, registered_date │ │ usage_type, tags[] │ │ preferences{} (Embedded) │ │ parts: [ObjectId × 8] ────►│───┐ │ orders[] (Embedded) │ │ total_price │ │ │ └── items[] (Embedded - 3 levels)│ │ compatibility_verified │ │ │ saved_builds: [ObjectId] ──────────┤ │ created_at, last_updated │ │ │ │ └──────────────────────────────┘ │ └─────────────────────────────────────┘ │ │ builds.parts ───────────┘ users.saved_builds → components._id → builds._id (Many-to-Many) (One-to-Few) users.orders.items.part_id → components._id (Reference inside Embedded)

כמות הנתונים: הקטלוג מכיל כ-46,000 רכיבים מנתוני אמת של PCPartPicker, מתוכם כ-10,000 עם מחיר עדכני. הרכיבים ללא מחיר נשמרים בכוונה כדי לדמות מצב מציאותי של מוצרים שאזלו מהמלאי.

14. אינדקסים (Indexes)

בפרויקט זה לא נוצרו אינדקסים מפורשים מעבר לאינדקס ברירת המחדל (_id). הסיבות:

  • דרישת הקורס: יצירת אינדקסים לא נדרשה בסילבוס.
